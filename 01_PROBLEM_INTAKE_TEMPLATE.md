# Problem Intake

**Use case name:** SOC Executive Dashboard (HACK-SOC-01)

**Who is the user?**
A multi-tenant MSSP (Managed Security Service Provider) SOC that serves many
customers across multiple partners. Detections come from each customer's own SIEM
(IBM QRadar "Offence" or Cortex XSIAM "Issue"), get turned into incident tickets in
the internal platform (SecurityHub) via playbook automation, and analysts triage,
open, work, and close them. Today there is no single place for leadership or
customers to see SOC performance — alert volume, detection/response speed, SLA
compliance, noise (false positives). The data exists in SecurityHub; it just isn't
surfaced.

**What's the core flow (one sentence)?**
A user logs in, lands on a single-pane dashboard scoped to what they're allowed to
see, sets global filters (customer, SIEM, SOAR, tier, time range), and reads 9 KPIs
+ two weekly trend charts that all move together off the same filter state, with
drill-down into any incident.

**Roles involved (for RBAC):**
- `super_admin` — sees all partners, all customers
- `partner_manager` — sees only their own partner's customers
- `customer_viewer` — sees only their own customer's data
- `analyst` — read-only, scoped to their own partner

**What does each role see/do differently?**
See the full RBAC Matrix in `02_SOLUTION_ARCHITECTURE_TEMPLATE.md`. In short: scope
narrows top-down (super_admin → partner_manager → customer_viewer), and `analyst`
is read-only within their partner (can't hit admin actions). Multi-tenant scope
enforcement is core to this use case, not incidental — a partner_manager token must
never be able to see another partner's data by editing a query parameter.

**Must-have for the demo:**
- Global filter bar (customer, SIEM, SOAR, tier, range) driving every card and chart
  from one shared state
- 9 KPI cards with week-over-week delta: Alerts, Critical Alerts, Incidents, Avg
  MTTD, Avg MTTR, SLA Compliance %, SLA Breaches, False-Positive Rate, P1/P2/P3 Avg
  Response
- Two trend charts: volume (alerts + incidents, weekly) and MTTR vs SLA
- Login as different roles live, showing tenant scope actually narrowing the data —
  this is the single most convincing 30 seconds of the demo
- Incident drill-down (MITRE techniques, assigned analyst, recommendation)

**Explicitly out of scope:**
- Real SIEM/SOAR integrations — data is generated by `backend/seed.py`, not pulled
  live from QRadar/XSIAM/XSOAR/Resilient
- Postgres/production database — SQLite for the hackathon (see `02` for the WHY and
  the `DB_TYPE` toggle that keeps a Mongo/Postgres migration path open)
- Multi-service deployment — logically separate services (`auth`, `incidents`,
  `analytics`, `customers`), physically one FastAPI process for build speed

**Data entities (rough shape):**
`users`, `customers`, `incidents` — full column-level schema is in
`02_SOLUTION_ARCHITECTURE_TEMPLATE.md`.
